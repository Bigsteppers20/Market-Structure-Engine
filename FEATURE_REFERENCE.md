# Feature Reference -- Market Structure Engine (post-optimization)

Complete, code-derived inventory of every dimension emitted by `MarketStructureEngine.feature_vector()` / `MarketState.to_dict()`, reflecting the feature-optimization pass described in `FEATURE_OPTIMIZATION_REPORT.md`. Generated by walking the current implementation feature-by-feature -- nothing here is inferred or invented.

**Verified feature count: 185** (matches `feature_vector()[0].shape[0]` observed on live OANDA EUR\_USD M5 and H1 test runs after optimization).

## Legend

- **Category**: Task 7 taxonomy (Trend / Momentum / Volatility / Liquidity / Price Action / Market Structure / Institutional Concepts / Volume / Session / Time / Statistical / Pattern Recognition / Microstructure).
- **Predictive Value**: High/Medium/Low + reason -- a domain-informed prior, not derived from any trained model (Task 8 explicitly excludes model-building).
- **Suggested Scaler**: one of StandardScaler / MinMaxScaler / RobustScaler / No Scaling Required, chosen from the feature's known range/units (Task 6). The engine itself performs **no normalization** -- this is guidance only.
- **Depends on historical candles only**: `Yes` means the feature is computed solely from candles at or before the point in the pipeline that consumes it.
- **Look-ahead bias check**: `No` = strictly causal/trailing computation. `No†` = correct for the single "as of last loaded candle" snapshot the engine produces, but computed via a full-series scan relative to an *earlier* bar's own index. See Data-Leakage Analysis.

## Data-Leakage Analysis (Task 11 re-audit)

**No leakage exists for the engine's intended usage** (call `load()` once with all history available up to "now", call `analyze()`, read only the final snapshot). This conclusion is unchanged by the optimization pass; the new spread features (`spread_*`) use only trailing rolling windows (`pandas.Series.rolling`, never `center=True`), so they introduce no new look-ahead risk. The two pre-existing structural nuances still apply:

1. **Swing confirmation lag** (not leakage, a delay): swings need `swing_window` bars *after* the pivot to confirm, so `trend_*`, `structure_*`, `sr_*`, `liq_*`, `micro_*` always describe the *last confirmed* structure, lagging the final bar.
2. **Full-series fill/mitigation scans** (leakage risk only if mis-replayed): `FvgEngine`/`OrderBlockEngine` compute fill/mitigation using the rest of the *loaded* series relative to each event's own index. Correct for a single snapshot; re-run `engine.load(df.iloc[:i+1]).analyze()` fresh per bar for any per-bar labeling/backtest use case.
3. **Cumulative VWAP** still accumulates from the first loaded row, not a session boundary -- unchanged, not a leakage risk, but a scale-consistency one.
